Continous Integration With Rails, RuboCop and Travis CI

Jun 21st, 2014 • Permalink

RuboCop is a great tool for checking the code quality in your Ruby project. It’s based on the community-authored Ruby Style Guide. :

It comes with a good set of defaults but can be customized to your own preferences with a .rubocop.yml file in your project folder.

Installation

You can install RuboCop globally with:

gem install rubocop

And then run it on your project with:

rubocop .

On an existing project, you will probably be flooded with warnings about syntax problems, so it’s best improve things gradually. Start with: rubocop --auto-gen-config, which creates a .rubocop_todo.yml. Then add inherit_from: .rubocop_todo.yml on the first line of .rubycop.yml. With this approach, you can gradually remove checks from .rubocop_todo.yml as your fix the problems with your code.

Rails Mode

RuboCop can also check for problems with Rails code:

rubocop . --rails

Integrating

Add gem 'rubocop' to your Gemfile. You can limit it to only the :development and :test groups since it’s not needed in production. Run bundle to update the Gemfile.lock.

gem 'rubocop', require: false ?

Next, create a new Rake task in lib/tasks/rubocop.rake

“`ruby if Rails.env.development? || Rails.env.test?

require "rubocop/rake_task"

desc "Run RuboCop"
Rubocop::RakeTask.new(:rubocop) do |task|
  task.options = ["--rails"]
end

end “`

You should now be able to run bundle exec rake rubocop. The advantage of this approach over installing globally is for when you’re in a team, so that you know everyone is running the same version.

By convention in Ruby projects, the rake command runs the tests for the projects. We can hook into this to run RuboCop too.

Edit your projects Rake and add the following line:

task spec: :rubocop

This defines the :spec task as dependent on the :rubcop task. If the RuboCop task fails, then the spec task will not run.

TravisCI

We could just print out the RuboCop warnings on the Travis log, but these are easy to miss or ignore.

Instead, we want to cause the build to fail.

This might seem drastic. And you might have to convince your team members. If they push without running rake first then the build will fail.

But by doing this you set habits in the team. They will start to write code which doesn’t trigger the.

I’m using Travis CI as an example, but this could easily be adapted for any other CI system.
